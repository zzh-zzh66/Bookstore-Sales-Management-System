# 书店销售管理系统 - 项目策划书

---

## 一、项目核心目标

### 1.1 项目背景
随着书店业务规模的增长，传统的人工管理方式已难以满足日常运营需求。本项目的目标是构建一套完整的书店销售管理系统，实现对图书入库、销售、会员管理及系统管理的全面信息化，提升书店运营效率与管理水平。

### 1.2 核心目标
| 编号 | 目标描述 | 优先级 |
|------|----------|--------|
| G-01 | 构建图书全生命周期管理能力，实现图书入库、库存追踪、信息维护的自动化 | P0 |
| G-02 | 建立高效的图书检索与统计体系，支持多维度条件查询与数据洞察 | P0 |
| G-03 | 实现销售全流程数字化管理，包括销售记录、实时库存扣减、多维度销售统计与报表生成 | P0 |
| G-04 | 搭建会员管理体系，支持会员申请、等级划分、差异化折扣策略 | P1 |
| G-05 | 建设完善的系统管理后台，涵盖参数配置、权限控制和安全管理 | P1 |
| G-06 | 确保系统具有良好的可扩展性、响应性能和数据一致性 | P1 |

### 1.3 预期成果
1. **可运行系统**：一个基于 B/S 架构、功能完整的书店销售管理系统，支持主流浏览器访问
2. **技术交付物**：
   - 完整的 MySQL 数据库脚本（含表结构、索引、初始数据）
   - Spring Boot 后端工程（含 RESTful API、业务逻辑、安全认证）
   - HTML5 前端页面（响应式界面、交互友好的操作体验）
3. **文档交付物**：
   - 项目需求规格说明书
   - 系统设计文档（含数据库设计、接口设计、架构设计）
   - 部署与运维手册
   - 用户操作手册

## 六、团队协作方式

### 6.1 角色分工

| 角色 | 人数 | 职责说明 |
|------|------|----------|
| 项目经理 | 1 人 | 项目整体协调、进度跟踪、风险管理 |
| 后端开发 | 2-3 人 | 数据库设计、API 开发、业务逻辑实现、系统安全 |
| 前端开发 | 1-2 人 | 页面 UI 实现、交互开发、数据可视化、接口联调 |
| 测试人员 | 1 人 | 测试用例编写、功能测试、性能测试、缺陷跟踪 |
| 产品/需求 | 1 人 | 需求分析、原型设计、验收标准制定 |

### 6.2 开发流程

```
需求分析 → 任务分解 → 编码实现 → 自测联调 → Code Review → 提测 → 验收
```

1. **每日站会**（15 分钟）：同步昨日完成、今日计划、遇到的问题
2. **代码评审**（Code Review）：每位开发者提交 PR 后需至少一人 Review 通过方可合并
3. **结对编程**（可选）：对复杂业务逻辑模块采用结对编程方式提高代码质量
4. **周例会**（30 分钟）：每周五下午进行进度同步、问题复盘、下周计划

### 6.3 版本控制与分支策略

```
main ──────── 生产分支，仅用于发布
  │
  ├── develop ──── 开发主分支，功能开发完成后合并至此
  │     │
  │     ├── feature/book-management ──── 图书管理功能分支
  │     ├── feature/sales-management ──── 销售管理功能分支
  │     ├── feature/member-management ──── 会员管理功能分支
  │     ├── feature/system-management ──── 系统管理功能分支
  │     └── feature/report-statistics ──── 统计报表功能分支
  │
  └── hotfix/xxx ──── 紧急修复分支
```

**分支命名规范**：`type/description`，如 `feature/book-crud`、`bugfix/inventory-error`

**提交信息规范**：
- `feat: 新增图书入库功能`
- `fix: 修复库存扣减并发问题`
- `docs: 更新 API 接口文档`
- `refactor: 重构销售统计查询逻辑`
- `test: 添加会员等级计算单元测试`

### 6.4 代码规范

**后端规范**：
- 包命名：`com.bookstore.{module}`（如 `com.bookstore.book`）
- 类名：大驼峰（`BookController`、`SaleService`）
- 方法名：小驼峰（`getBookById`、`createSaleOrder`）
- API 路径遵循 RESTful 风格：`/api/v1/books`、`/api/v1/sales`
- 统一返回结果格式：`{ code, message, data }`

**前端规范**：
- 文件命名：小写字母 + 连字符（`book-list.html`、`member-form.html`）
- CSS 类名：小写 + 连字符（`.book-table`、`.sale-form-group`）
- JavaScript 函数：小驼峰（`loadBookList`、`submitSaleOrder`）
- 页面文件按模块组织目录结构

## 八、质量保障措施

1. **代码质量**：
   - 后端使用 SonarQube 进行代码质量扫描
   - 前后端代码均需通过 Code Review
   - 关键业务逻辑编写单元测试

2. **接口质量**：
   - 使用 Swagger 生成并维护 API 文档
   - 前后端依据接口文档独立开发联调
   - API 版本管理，向后兼容

3. **数据质量**：
   - 数据库字段约束（NOT NULL、UNIQUE、外键）
   - 后端参数校验（JSR 303 Bean Validation）
   - 事务管理保证数据一致性

4. **安全性**：
   - 密码加密存储（BCrypt）
   - JWT 令牌认证
   - 接口权限校验（角色 + 权限）
   - 防 SQL 注入（MyBatis 参数绑定）
   - 防 XSS 攻击（输入过滤 + 输出编码）
